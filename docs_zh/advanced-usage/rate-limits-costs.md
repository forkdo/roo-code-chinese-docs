# 速率限制与成本

了解并管理 API 使用情况对于流畅且经济高效地使用 Roo Code 至关重要。本节介绍如何跟踪令牌使用情况和成本。速率限制默认为 0（已禁用），通常无需调整，现在按配置文件进行配置；如需设置，请参阅 [API 配置文件](/features/api-configuration-profiles#creating-a-profile) 文档了解详细信息。

---

## 令牌使用情况

Roo Code 使用令牌与 AI 模型进行交互。令牌本质上是单词的组成部分。请求和响应中使用的令牌数量会影响处理时间和成本。

*   **输入令牌：** 这些是提示中的令牌，包括系统提示、您的指令以及提供的任何上下文（例如文件内容）。
*   **输出令牌：** 这些是 AI 模型在其响应中生成的令牌。

您可以在聊天历史记录中查看每次交互使用的输入和输出令牌数量。

---

## 成本计算

大多数 AI 提供商根据使用的令牌数量收费。定价因提供商和具体模型而异。

Roo Code 会根据配置的模型定价自动计算每次 API 请求的预估成本。此成本显示在聊天历史记录中，与令牌使用情况并列。

对于具备推理能力的模型（例如 Gemini 3 Pro Preview 以及其他公开单独的“思考”或推理令牌的模型），当提供商报告时，Roo Code 现在会在预估中同时包含普通令牌 **和** 推理/“思考”令牌。这可能会使显示的令牌使用量和成本略高于旧版本，但它能更好地匹配提供商的实际计费方式。

**注意：**

*   成本计算仅为*预估*。实际成本可能因提供商的计费方式而略有不同。
*   某些提供商可能提供免费套餐或积分。请查看提供商的文档了解详细信息。
*   某些提供商提供提示缓存功能，可大幅降低成本。

### 限制自动批准请求

为了进一步帮助管理 API 成本并防止意外支出，Roo Code 为自动批准操作包含了一个“最大请求数”设置。这允许您定义 Roo Code 在任务期间无需您明确重新批准即可连续进行的 API 调用的具体限制。

*   **工作原理：** 如果您设置了限制（例如 5 个请求），Roo Code 将执行最多 5 次自动批准的 API 调用。在进行第 6 次调用之前，它将暂停并提示您“重置并继续”，如下图所示。
    <img src="/img/v3.18.0/v3.18.0-1.png" alt="警告消息，指示已达到自动批准请求限制。" width="600" />
    *达到自动批准请求限制时的通知。*
*   **配置：** 此限制在“自动批准操作”设置中进行配置。您可以设置一个具体数字或选择“无限制”。有关配置此设置及其他自动批准设置的详细步骤，请参阅 [自动批准操作文档](/features/auto-approving-actions)。
    <img src="/img/v3.18.0/v3.18.0.png" alt="在 Roo Code 设置中设置自动批准操作的最大请求数限制。" width="600" />
    *设置自动批准操作的“最大请求数”。*

此功能提供了一个额外的保障，尤其是在涉及多个 API 调用的复杂或长时间运行的任务中。

## 优化令牌使用情况的技巧

*   **保持简洁：** 在提示中使用清晰简洁的语言。避免不必要的词语或细节。
*   **仅提供相关上下文：** 有选择性地使用上下文提及 (`@file.ts`, `@folder/`)。仅包含与任务直接相关的文件。
*   **分解任务：** 将大型任务分解为更小、更专注的子任务。
*   **使用自定义指令：** 提供自定义指令以指导 Roo Code 的行为，减少在每个提示中进行冗长解释的需要。
*   **选择合适的模型：** 某些模型比其他模型更具成本效益。对于不需要大型模型全部功能的任务，请考虑使用更小、更快的模型。
*   **使用模式：** 不同的模式可以访问不同的工具，例如 `Architect` 无法修改代码，这使得它在分析复杂代码库时成为一个安全的选择，无需担心意外允许昂贵的操作。
*   **如果未使用，请禁用 MCP：** 如果您不使用 MCP（模型上下文协议）功能，请考虑 [在 MCP 设置中禁用它](/features/mcp/using-mcp-in-roo#enabling-or-disabling-mcp-server-creation)，以显著减小系统提示的大小并节省令牌。

通过了解并管理您的 API 使用情况，您可以高效且有效地使用 Roo Code。
